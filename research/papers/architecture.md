# Architecture of a Database System (Hellerstein, Stonebraker, Hamilton, 2007)

Source: J. M. Hellerstein, M. Stonebraker, J. Hamilton. "Architecture of a Database System." Foundations and Trends in Databases, Vol. 1, No. 2 (2007), pp. 141-259. DOI: 10.1561/1900000002. The extracted text used for this note covers Sections 1 through 4. Page and section citations below refer to that text.

This note covers chapters 1-4 only: Introduction and the five components (Section 1), Process Models and admission control (Section 2), Parallel Architecture (Section 3), and the Relational Query Processor (Section 4). Storage, transactions, and shared utilities (Sections 5-7) are out of scope here.

## 1. What problem this paper solves and why it mattered

The paper fills a documentation gap. Textbooks teach the algorithms and theory of databases (relational algebra, B+-trees, the System R optimizer) but skip the systems engineering that makes a real DBMS run as a multi-user server. As the abstract puts it, "there has been relatively sparse coverage in the literature of the systems design issues that make a DBMS work" (p. 141). The authors note three reasons the knowledge is not widely held: the applied DBMS community is small with only a handful of successful implementations, academic treatment ignores architectural issues in favor of what is "natural to teach, study, and test," and much of the wisdom lives in product manuals or the oral tradition of the community (Section 1, pp. 142-143). It mattered because DBMSs were among the earliest widely deployed online server systems and pioneered scalability and reliability techniques later reused across operating systems, applications, and networked services (Section 1, p. 142). The paper is grounded in real commercial and open-source systems (DB2, Oracle, SQL Server, PostgreSQL, MySQL, Informix, Sybase) and explicitly contrasts the different design choices those vendors made.

## 2. The core mechanism, from first principles

### 2.1 The five components and the life of a query (Section 1.1)

A typical RDBMS has five main components (Figure 1.1, p. 144). The paper introduces them by tracing a single-query transaction: a gate agent at an airport clicks a form to request the passenger list for a flight (p. 144).

1. Client Communications Manager. Establishes and remembers connection state for the caller (client or middleware), responds to SQL commands, and returns data and control messages. The paper notes two-tier (client-server, e.g., ODBC/JDBC), three-tier (with a middle-tier web server or TP monitor), and four-tier (adding an application server) deployments, so a DBMS must speak many connectivity protocols (pp. 144-145).
2. Process Manager. Assigns a "thread of computation" to the command and wires its data and control outputs back through the communications manager. The most important decision here is admission control: whether to begin the query now or defer it until resources are available (pp. 145-146).
3. Relational Query Processor. Checks authorization, compiles SQL text into an internal query plan, and runs that plan through the plan executor, which is a suite of operators (join, selection, projection, aggregation, sort) plus operators that fetch records from below (p. 146).
4. Transactional Storage Manager. Manages all data access and manipulation through access methods (heaps, indexes), the buffer manager, the lock manager (for ACID isolation), and the log manager (for durability and rollback) (pp. 146-147).
5. Shared components and utilities (right side of Figure 1.1). The catalog manager and memory manager are invoked during essentially every query; the catalog is used during authentication, parsing, and optimization. Other utilities (replication, loaders, backup, monitoring) run independently of any query (p. 147).

The result is delivered by "unwinding the stack": access methods return control to executor operators, which build result tuples and place them in a buffer for the communications manager to ship back. Large result sets cause repeated FETCH iterations through communications manager, executor, and storage manager (pp. 146-147).

### 2.2 Process models (Section 2)

The chapter defines its vocabulary precisely (Section 2, pp. 149-152):
- An OS process couples a thread of control with a private address space, OS resource handles, and security context, scheduled by the kernel.
- An OS thread (kernel thread, k-thread) is a kernel-scheduled execution unit with no private address space; threads in the same process share memory.
- A lightweight thread package is an application-level construct scheduled in user space. It switches faster than kernel threads (no kernel mode switch), but any blocking operation by one thread blocks all threads in the process, so LWT packages must use only asynchronous, non-blocking I/O and avoid blocking OS calls.
- DBMS threads are a DBMS's own LWT package.
- A DBMS Client implements the application API (JDBC, ODBC, OLE/DB) and marshals calls to the server.
- A DBMS Worker is the thread of execution doing work for a client. There is a 1:1 mapping between a worker and a client: the worker handles all SQL from that one client.

Under two simplifying assumptions (good OS thread support and uniprocessor hardware), there are three natural process models (Section 2.1, p. 152):

1. Process per DBMS worker (Section 2.1.1). Each worker maps to an OS process. Simple to build, gets OS protection and tooling (debuggers, memory checkers) for free, but scales worst because a process has more state and process switches cost more (security context, memory manager state, file/network handle tables). Shared structures (lock table, buffer pool) must live in OS shared memory, which erodes the address-space-isolation benefit. Used by DB2, PostgreSQL, Oracle (pp. 153-154).
2. Thread per DBMS worker (Section 2.1.2). One multithreaded process; a dispatcher thread accepts connections and allocates a thread per connection. Scales well to many connections but inherits multithreaded hazards (no memory protection between threads, hard race-condition debugging, porting friction across thread APIs). Used by DB2, SQL Server, MySQL, Informix, Sybase (pp. 154-155).
3. Process pool (Section 2.1.3). A variant of process per worker. A bounded, often dynamically resizable pool of processes services requests; each SQL statement runs to completion on a borrowed process which is then returned. More memory efficient than process per worker; if all processes are busy a new request waits (pp. 155-156).

Shared data crosses process boundaries through buffers (Section 2.1.4, pp. 156-159). The two major I/O buffer types are disk I/O buffers (the buffer pool for database pages, and the log tail for log records) and client communication buffers. In thread-per-worker these are heap-resident; in the other two models they live in shared memory. The log tail is an in-memory FIFO queue flushed to log disk(s). A transaction cannot be reported committed until its commit log record is flushed (the commit flush), and the server holds locks until then; group commit batches multiple commit records into one I/O. The lock table is shared by all workers and used by the lock manager.

DBMS threads (Section 2.2) exist because efficient, scalable OS threads were unavailable when these systems were designed: OS threads were not widely implemented until the 1990s and even today some implementations do not scale well. So for legacy, portability, and scalability reasons many DBMSs built their own LWT package, accepting that this replicates OS logic (task switching, scheduling, thread state) inside the DBMS (pp. 159-160).

Standard practice (Section 2.3, pp. 160-162): DB2 supports four distinct process models and picks based on OS thread quality (thread per worker or thread pool where threads are good; process per worker or process pool where they are not). PostgreSQL runs process per worker exclusively. SQL Server defaults to workers multiplexed over a thread pool, with over 99% of installations running that way, and offers DBMS-threads-on-OS-threads as a non-default "Fibers" option used mainly in high-scale TPC benchmarks. Oracle defaults to process per worker and offers process pool. Intra-query parallelism is the temporary assignment of multiple workers to one query; it does not change the underlying process model.

Admission control (Section 2.4, pp. 162-164) prevents thrashing. As load rises, throughput climbs to a maximum then collapses as the system thrashes, usually from memory pressure when the working set of pages no longer fits in the buffer pool (sorting and hash joins are the worst offenders), and sometimes from lock contention causing repeated deadlock-rollback-restart. A good admission controller produces graceful degradation: latency rises proportionally with arrival rate while throughput stays at peak. It is done in two tiers: a simple dispatcher-level check that caps client connections (sometimes omitted and left to a higher tier), and an execution admission controller inside the relational query processor that runs after parse and optimize and decides whether to postpone the query, start it with fewer resources, or start it unconstrained. It uses optimizer estimates of disk I/O per device, CPU load, and most importantly memory footprint (including sort and hash space), since memory pressure is the main cause of thrashing.

### 2.3 Parallel architecture (Section 3)

The three canonical hardware models, using terminology from DeWitt and Gray (reference [87]):

- Shared memory (Section 3.1, pp. 165-167). All processors access the same RAM and disk at roughly equal cost. Standard server hardware (commonly two to eight processors). Multi-core chips share caches and the memory bus, so they look like shared-memory to the programmer. All three process models from Section 2 run well here; most database systems evolved from uniprocessor to shared-memory implementations, and the OS transparently assigns workers across processors. The remaining challenge is parallelizing a single query across CPUs.
- Shared nothing (Section 3.2, pp. 167-169). A cluster of independent machines over a network; no machine can directly touch another's memory or disk. Tables are horizontally partitioned across nodes (hash, range, round-robin, or hybrid). Each node accesses, locks, and logs its own local data; the optimizer chooses how to re-partition tables and intermediate results and assigns each machine a logical partition of work; executors ship data requests and tuples but never thread state. Minimal coordination is required, but good partitioning is critical, which burdens the DBA and the optimizer. Cross-processor coordination is still needed for transaction completion, load balancing, distributed deadlock detection, and two-phase commit. Partial failure must be handled: bring all nodes down (emulating shared memory), Informix "Data Skip" (run on live nodes, skipping failed-node data, with no well-defined semantics), or redundancy schemes from full failover to chained declustering. Under chained declustering the n-1 surviving nodes each take n/(n-1) of the original load, giving linear degradation. Unbeatable scalability and cost; used mostly at the high end for decision support and warehouses; a shared-nothing cluster is often built from shared-memory nodes.
- Shared disk (Section 3.3, pp. 170-171). All processors reach the disks at roughly equal cost but cannot read each other's RAM. Examples: Oracle RAC and DB2 for zSeries SYSPLEX; popularized by SANs. Lower administration cost (no mandatory partitioning) and a single node's failure does not block others' access to the whole database. But there is no natural place to coordinate sharing since each machine has its own locks and buffer pool, so it needs a distributed lock manager and a cache-coherency protocol, which are complex and can bottleneck under contention. zSeries SYSPLEX implements the lock manager in hardware. Shared disk still has single points of failure: a page corrupted before reaching storage is visible (corrupt) to all nodes, and RAID will faithfully replicate the corruption.

NUMA (Section 3.4, pp. 171-172) gives a shared-memory programming model over independent memories with non-uniform access times (local fast, remote slower). NUMA clusters never won commercial share, but the NUMA design pattern is now common inside large shared-memory multiprocessors, where memory is split into "pods" of CPUs plus local memory. The rule of thumb: when the near-to-far access ratio exceeds about 1.5:1 to 2:1, the DBMS must optimize by (a) allocating memory local to the processor and (b) scheduling a worker on the same processor it ran on before.

DBMS threads on multiprocessors (Section 3.5, pp. 172-173): a single OS process runs on only one processor at a time, so a DBMS-thread package living in one process leaves the rest of a multiprocessor idle. Early Sybase suffered this. The fix is to run DBMS threads across multiple OS processes plus thread migration between processes for load balancing (Informix did this well from Version 6.0). Rule of thumb: one process per physical processor, to maximize hardware parallelism while minimizing per-process memory overhead.

Standard practice (Section 3.6, pp. 173-174): all major vendors support shared-memory parallelism. Shared nothing: DB2, Informix, Tandem, Teradata, and Greenplum (a custom PostgreSQL). Shared disk: Oracle RAC, RDB, DB2 for zSeries. No leading commercial system supports both shared-nothing and shared-disk in one code base, and SQL Server (as of the paper) implemented neither cluster model.

### 2.4 Relational query processor (Section 4)

A relational query processor "takes a declarative SQL statement, validates it, optimizes it into a procedural dataflow execution plan, and (subject to admission control) executes that dataflow program," after which the client pulls result tuples one at a time or in small batches (Section 4, p. 176). Query processing is treated as a single-user, single-threaded task; concurrency control is handled transparently by lower layers, except that the executor must pin and unpin buffer pool pages during brief critical operations. DML (SELECT, INSERT, UPDATE, DELETE) goes through the optimizer; DDL (CREATE TABLE, CREATE INDEX) is usually handled procedurally by static logic, not the optimizer (Section 4, pp. 176-177).

Parsing and authorization (Section 4.1, pp. 177-178). The parser checks correctness, resolves names and references, converts the query to the optimizer's internal format, and verifies authorization. It canonicalizes table names to a four-part name (server.database.schema.table), resolves table aliases, calls the catalog manager to confirm tables exist and attribute references are valid, and uses attribute data types to disambiguate overloaded operators and constants (the worked example is `(EMP.salary * 1.15) < 75000`, whose code path depends on whether salary is integer, float, or money). Authorization can be partly deferred to execution time, which is required for row-level (data-value-dependent) security, and even when static checking is possible, deferring lets plans be shared across users and avoids recompilation when permissions change.

Query rewrite (Section 4.2, pp. 179-182). The rewriter simplifies and normalizes a query without changing its semantics, using only the query and catalog metadata, never table data. Its responsibilities:
- View expansion: replace each view reference in FROM with the view's tables and predicates, recursively, until no views remain. This technique was first proposed for the set-based QUEL language in INGRES.
- Constant arithmetic evaluation: `R.x < 10+2+R.y` becomes `R.x < 12+R.y`.
- Logical rewriting of predicates: e.g., `NOT Emp.Salary > 1000000` becomes `Emp.Salary <= 1000000` to match index access methods; unsatisfiable conjunctions like `salary < 75000 AND salary > 1000000` collapse to FALSE (enabling empty results without disk access and "partition elimination" in range-partitioned SQL Server); predicate transitivity adds new predicates (`R.x < 10 AND R.x = S.y` implies `S.y < 10`) so the optimizer can filter earlier.
- Semantic optimization: use integrity constraints, e.g., redundant join elimination removes a foreign-key-referenced table (Dept) when its columns are not used and the foreign key is non-null.
- Subquery flattening and heuristic normalization: because optimizers usually work on one SELECT-FROM-WHERE block at a time, the rewriter flattens nested and correlated subqueries to expose single-block optimizations. Flattening correlated subqueries is especially important in parallel architectures because correlation forces nested-loop, serialized execution.

Query optimizer (Section 4.3, pp. 182-188). The optimizer turns the internal representation into an efficient plan, viewable as a dataflow graph piping table data through operators (Figure 4.1). Each SELECT-FROM-WHERE block is optimized with techniques from Selinger et al.'s System R optimizer (reference [79]), post-processing operators are added on top of each block for GROUP BY, ORDER BY, HAVING, DISTINCT, and the blocks are stitched together. On plan representation: the original System R prototype compiled plans to machine code while early INGRES generated an interpretable plan. INGRES called interpretation a "mistake" in retrospect, yet Moore's law vindicated it; conversely System R people called machine-code compilation a mistake, and the first change when System R became commercial (SQL/DS) was to replace the machine-code executor with an interpreter. Every major DBMS now compiles to some interpretable structure, differing only in abstraction level (algebra-like vs. low-level op-codes like Java byte codes).

Selinger's paper is "the bible of query optimization" but was "preliminary research," extended by all systems along these dimensions:
- Plan space: System R considered only left-deep plans (the right input to a join must be a base table) and postponed Cartesian products; modern systems also consider bushy trees and early Cartesian products in some cases.
- Selectivity estimation: System R used simple table and index cardinalities; modern systems use histograms and summary statistics, sometimes built by sampling to avoid full scans, and join selectivity is estimated by joining histograms on join columns. Ioannidis and Christodoulakis (reference [45]) showed that selectivity errors early in optimization propagate multiplicatively up the plan tree. A benchmark flaw is noted: TPC-D and TPC-H generated statistically independent column values, discouraging adoption of techniques for real, correlated distributions; TPC-DS fixed this.
- Search algorithms: Microsoft and Tandem replaced Selinger's bottom-up dynamic programming with goal-directed top-down search based on Cascades (reference [25]). Both approaches work well in practice; both still have runtime and memory exponential in the number of tables. Systems fall back on proprietary heuristics for queries with too many tables (MySQL's optimizer was at the time entirely heuristic, leaning on indexes and key/foreign-key constraints), and some require optimizer hints for very large FROM clauses.
- Parallelism: Hong and Stonebraker (reference [42]) used a two-phase scheme, first picking the best single-system plan then scheduling it across processors or machines, to dodge the complexity of one-pass parallel optimization. Oracle OPS/RAC uses two-phase; DB2 Parallel Edition started two-phase and has been moving toward single-phase. The authors call this area "more like art than science."
- Auto-tuning: workload-driven "what-if" analyses (Chaudhuri and Narasayya, reference [12]) and the LEO Learning Optimizer (Markl et al., reference [57]).

Compilation and recompilation (Section 4.3.1, pp. 187-188). SQL lets a query be prepared (parsed, rewritten, optimized, plan stored) and later executed, even with program variables in place of constants, with the caveat that selectivity estimation assumes "typical" values for those variables, which can yield poor plans when atypical. Dynamically built SQL (e.g., Ruby on Rails) cannot be precompiled, so DBMSs keep a query plan cache and reuse cached plans for the same or similar statements. Plans must be re-optimized over time: at minimum, dropping an index invalidates plans that used it. Vendors differ philosophically. IBM favors predictable performance and re-optimizes only when a plan can no longer run (e.g., dropped index); Microsoft favors self-tuning and re-optimizes aggressively (a significant cardinality change triggers recompilation in SQL Server). This traces to customer base: IBM's high-budget shops with skilled DBAs want stability; Microsoft's low-end entry wanted the DBMS to tune itself.

Query executor (Section 4.4, pp. 188-191). The executor operates on a fully specified plan, typically a directed dataflow graph of operators. Most modern executors use the iterator model from the earliest relational systems. The iterator superclass (Figure 4.2) is:

```
class iterator {
  iterator &inputs[];
  void init();
  tuple get_next();
  void close();
}
```

Every operator (filescan, indexscan, sort, nested-loops join, merge-join, hash-join, duplicate-elimination, grouped-aggregation) is a subclass. The key property: any iterator subclass can feed any other, so each operator's logic is independent of its parents and children, and no special-case code is needed for particular combinations (Graefe's survey, reference [24], gives more detail; PostgreSQL is cited as readable reference code).

Iterator discussion (Section 4.4.1): iterators couple dataflow with control flow. `get_next()` is an ordinary procedure call returning a tuple reference via the call stack, so a tuple reaches a parent exactly when control returns. Consequences: one DBMS thread can execute the whole graph, no queues or rate-matching between operators are needed, and the design is clean to implement and debug (contrast with network dataflow, which needs queuing protocols). It is efficient for single-system execution: since one iterator is always active, resource utilization (and thus time to completion) is maximized. Optimization goals other than time to completion exist: throughput, and time to first row for interactive use. Parallelism needs essentially no change to the model: parallelism and network communication are hidden inside special exchange iterators (Graefe, reference [23]) that push data internally while preserving the pull-style `get_next()` API. The model can be inefficient only when iterators block on I/O and prefetching fails, which is rare in single-system databases but frequent over remote tables or clusters.

Where's the data (Section 4.4.2): the iterator superclass never allocates memory dynamically. Each iterator is pre-allocated a fixed number of tuple descriptors (one per input plus one output); a tuple descriptor is an array of column references, each a pointer to a tuple elsewhere plus a column offset. Tuples live in one of two places: BP-tuples reside in buffer pool pages, and referencing one requires incrementing the page's pin count (a count of active references to tuples on that page) and decrementing it when the descriptor clears; M-tuples are allocated on the memory heap by copying columns out of the buffer pool (the copy bracketed by a pin/unpin pair) or by evaluating expressions. Always copying into M-tuples simplifies the executor and avoids buffer leaks (forgetting to unpin), but memory copies are a serious bottleneck. Holding a BP-tuple pins a whole page, consuming buffer memory and constraining replacement. The efficient approach supports tuple descriptors that reference both kinds.

Data modification (Section 4.4.3, pp. 191-193): INSERT, DELETE, UPDATE plans are usually straight-line, with an access method as source and a modification operator at the end. The Halloween problem (discovered on October 31 by the System R group, and explicitly unrelated to the phantom problem) arises when a plan both reads and updates the same data: pipelining an index scan over Emp.salary into an update for "give everyone under $20K a 10% raise" can rediscover a tuple that moved rightward in the B+-tree after its raise, raising it repeatedly until it exceeds $20K. SQL semantics forbid a statement seeing its own updates. Fixes: have the optimizer avoid indexes on the updated column (safe but sometimes inefficient), or use a batch read-then-write scheme that materializes all target Record-IDs first, then fetches and updates them (right side of Figure 4.3); pipelined update schemes need multiversion storage support.

Access methods (Section 4.5, pp. 193-196). Access methods manage the disk-based structures: unordered files ("heaps") and indexes. All major systems implement heaps and B+-tree indexes; Oracle and PostgreSQL also offer hash indexes for equality lookups; some support R-trees; PostgreSQL has the extensible Generalized Search Tree (GiST) used for R-trees and RD-trees; DB2 UDB v8 added Multi-Dimensional Clustered (MDC) indexes; warehouse systems add bitmap index variants. The access method API is an iterator API, with `init()` extended to accept a search argument (SARG) of the form column-operator-constant; a NULL SARG means scan all tuples, and `get_next()` returns NULL when no tuple satisfies the SARG. There are two reasons to push SARGs into the access method layer: B+-trees need them to function efficiently, and (subtler, applying even to heap scans) evaluating the SARG one page at a time inside the access method avoids the per-tuple cost of returning a pinned tuple (or copy), having the caller test and reject it, then decrement the pin (or delete the copy) and re-invoke. Doing this a page at a time keeps a clean boundary between storage and relational engines while saving large amounts of CPU when streaming millions of tuples; this is the standard DBMS wisdom of amortizing work across a collection, here for CPU rather than disk. Row pointers: many DBMSs use physical RIDs (disk addresses), which are fast but make row movement expensive because every secondary index pointing at the row must be updated; rows move when an update grows them past the page's free space or when a B+-tree splits. DB2 uses a forwarding pointer (extra I/O to find a moved row, but no secondary-index update) and does not support B+-trees as primary storage for base tuples; SQL Server and Oracle use B+-trees as primary storage and put the primary key (plus uniqueness bits) in secondary indexes instead of a physical RID, trading some lookup speed to tolerate movement, with Oracle keeping a physical pointer alongside the key as a fast path. Oracle also lets rows span pages to avoid moving them. Access methods, unlike other iterators, interact deeply with concurrency and recovery logic.

## 3. Key facts and figures

- A typical RDBMS has five components: client communications manager, process manager, relational query processor, transactional storage manager, and shared components/utilities (Figure 1.1, p. 144).
- There is a 1:1 mapping between a DBMS worker and a DBMS client; the worker handles all SQL from that one client (Section 2, p. 151).
- Three baseline process models: process per worker, thread per worker, process pool (Section 2.1, p. 152).
- Process per worker is used by DB2, PostgreSQL, Oracle; thread per worker by DB2, SQL Server, MySQL, Informix, Sybase (Sections 2.1.1-2.1.2).
- DB2 supports four distinct process models; PostgreSQL uses process per worker exclusively; SQL Server defaults to thread-pool multiplexing with over 99% of installs running that way; its DBMS-threads-on-OS-threads option is called Fibers (Section 2.3, pp. 160-162).
- OS threads were not widely implemented until the 1990s, motivating DBMS-built lightweight thread packages (Section 2.2.1, p. 159).
- A lightweight thread package switches faster than kernel threads but a single blocking I/O blocks all threads; it must use only async non-blocking I/O (Section 2, pp. 150-151).
- Admission control yields graceful degradation: latency rises with arrival rate, throughput stays at peak; thrashing is usually from buffer-pool memory pressure (sorting, hash joins), sometimes from lock contention (Section 2.4, pp. 162-163).
- Admission control has two tiers: dispatcher-level connection cap and an execution admission controller inside the query processor that runs after parse and optimize, keyed mainly on memory footprint (Section 2.4, pp. 163-164).
- Three parallel hardware models: shared memory, shared nothing, shared disk; terminology from reference [87]. Typical servers ship 2 to 8 processors (Section 3.1, p. 165).
- Shared nothing partitions tables horizontally (hash, range, round-robin, hybrid) and needs two-phase commit and distributed deadlock detection across nodes (Section 3.2, pp. 167-168).
- Chained declustering: after one failure each of the n-1 survivors does n/(n-1) of the original work, giving linear degradation (Section 3.2, p. 169).
- Shared disk needs a distributed lock manager and a cache-coherency protocol; zSeries SYSPLEX implements the lock manager in hardware. Examples: Oracle RAC, DB2 for zSeries (Section 3.3, pp. 170-171).
- NUMA optimization threshold: near-to-far memory access ratio above roughly 1.5:1 to 2:1 (Section 3.4, p. 172).
- Multiprocessor rule of thumb: one OS process per physical processor (Section 3.5, p. 173).
- No leading commercial system supports both shared-nothing and shared-disk in one code base; SQL Server (as of 2007) implemented neither (Section 3.6, p. 174).
- A four-part canonical table name is server.database.schema.table (Section 4.1, p. 177).
- System R optimizer (Selinger et al., reference [79]) used left-deep plans, postponed Cartesian products, simple cardinality-based selectivity, and bottom-up dynamic programming (Section 4.3, pp. 182-184).
- Selectivity errors propagate multiplicatively up the plan tree (Ioannidis and Christodoulakis, reference [45]) (Section 4.3, p. 185).
- Top-down search alternative is based on Cascades (reference [25]); both DP and top-down search remain exponential in the number of tables (Section 4.3, p. 185).
- The iterator interface is init(), get_next(), close(), with an inputs array (Figure 4.2, p. 189). Iterators couple dataflow and control flow; one thread runs the whole graph (Section 4.4.1, pp. 189-190).
- Parallelism is hidden inside exchange iterators (Graefe, reference [23]) that preserve the pull-style get_next() API (Section 4.4.1, p. 190).
- Two in-flight tuple forms: BP-tuples (in the buffer pool, require pin count management) and M-tuples (on the heap) (Section 4.4.2, pp. 190-191).
- The Halloween problem was found on October 31 by the System R group; it is distinct from the phantom problem (Section 4.4.3, p. 192).
- Access methods expose an iterator API with a SARG (column-operator-constant), a System R term; NULL SARG means full scan (Section 4.5, p. 194).
- All major systems implement heaps and B+-tree indexes; PostgreSQL has the extensible GiST (reference [39]) (Section 4.5, pp. 193-194).

## 4. Trade-offs and limitations

- Process per worker vs thread per worker: process per worker gets OS isolation and tooling but scales worst (more per-process state, heavier context switches) and still needs shared memory for the lock table and buffer pool, which undercuts isolation. Thread per worker scales to many connections but loses inter-thread memory protection, makes debugging race conditions hard, and is harder to port across thread APIs (Sections 2.1.1-2.1.2).
- DBMS-built lightweight threads buy fast task switching and portability but force the DBMS to reimplement scheduling, thread-state management, and async I/O, with high implementation cost, poor tool support, and long-term maintenance burden (Sections 2.2.1, 2.3).
- Shared nothing: best scalability and cost, but partitioning quality is critical and burdens the DBA and optimizer, and partial failure has no clean answer (full shutdown, semantically murky Data Skip, or costly redundancy) (Section 3.2).
- Shared disk: easier administration and node-failure tolerance, but distributed lock manager and cache-coherency protocols are complex and can bottleneck under contention, and a pre-storage page corruption is visible (and replicated) to all nodes (Section 3.3).
- Plan representation: machine-code compilation (System R) vs interpretation (INGRES) was each called a mistake by the other camp; portability pushed every modern system to interpretable structures (Section 4.3).
- Optimizer search is exponential in the number of tables for both dynamic programming and top-down search; large FROM clauses fall back on proprietary heuristics or require hints (Section 4.3).
- Selectivity estimation from independent single-column statistics misstates correlated real data, and errors compound up the tree; column-dependency techniques were slow to adopt, partly due to the TPC-D/TPC-H independence flaw (Section 4.3).
- Prepared/cached plans risk poor performance when "typical" placeholder values are unrepresentative; re-optimization trades predictable performance (IBM) against self-tuning efficiency (Microsoft) (Section 4.3.1).
- M-tuples simplify executor code and avoid buffer leaks but cost memory copies; BP-tuples avoid copies but pin pages and constrain buffer replacement (Section 4.4.2).
- The iterator model can stall when an operator blocks on I/O and prefetching fails, common over remote or clustered tables (Section 4.4.1 footnote).
- Physical RIDs make lookups fast but row movement expensive; primary-key references in secondary indexes tolerate movement but slow secondary-index access (Section 4.5).
- The paper itself is scoped to architecture, not algorithmics, and explicitly omits many product-specific exceptions and finer nuances of multi-million-line code bases (Section 1.2).

## 6. Viva question bank

1. Q: Name the five components of an RDBMS and what each does. A: Client communications manager (connection state and protocol handling), process manager (assigns workers and does admission control), relational query processor (authorization, compilation, execution), transactional storage manager (access methods, buffer pool, lock and log managers for ACID), and shared components/utilities (catalog, memory manager, replication, backup). They are introduced in Section 1.1 via the gate-agent query and shown in Figure 1.1.

2. Q: What is the mapping between DBMS clients and DBMS workers, and why does it matter for the process models? A: It is 1:1: one worker handles all SQL from one client (Section 2, p. 151). It matters because the three process models differ only in how that worker is realized, as an OS process, an OS thread, or a borrowed process from a pool, which changes memory footprint, isolation, and scalability.

3. Q: Contrast process per worker with thread per worker. Which scales better and why? A: Thread per worker scales better to many connections because a thread carries less state than a process and thread switches avoid switching security context, memory-manager state, and file/network handle tables. Process per worker is simpler and gets OS memory protection and tooling, but consumes more memory and switches more slowly (Sections 2.1.1-2.1.2). Both still share the buffer pool and lock table, so process isolation is partial.

4. Q: Why did many DBMSs build their own lightweight thread packages, and what is the main hazard of lightweight threads? A: Efficient, scalable OS threads were not widely available until the 1990s and varied across platforms, so for legacy, portability, and scalability reasons DBMSs implemented their own (Section 2.2.1). The main hazard is that one blocking operation blocks every thread in the process, so the package must use only asynchronous non-blocking I/O and avoid blocking OS calls, replicating OS scheduling logic inside the DBMS.

5. Q: What is admission control, why is it needed, and where does it sit? A: It refuses new work unless resources are available, preventing thrashing so the system degrades gracefully (latency rises with arrival rate but throughput stays at peak) (Section 2.4). It sits in two tiers: a dispatcher-level cap on connections and an execution admission controller inside the query processor that runs after parse and optimize and keys mainly on the optimizer's estimated memory footprint, since buffer-pool memory pressure is the usual cause of thrashing.

6. Q: Distinguish shared memory, shared nothing, and shared disk. A: Shared memory: all processors reach the same RAM and disk at equal cost (typical 2 to 8 way servers and multi-core). Shared nothing: a cluster where no node touches another's RAM or disk, with tables horizontally partitioned. Shared disk: all processors reach the disks equally but cannot read each other's RAM (Sections 3.1-3.3). The terminology comes from reference [87].

7. Q: In shared nothing, what coordination is still required despite "minimal coordination," and how is partial failure handled? A: Even though value-based horizontal partitioning lets nodes work independently, they must still exchange control messages for transaction completion, load balancing, distributed deadlock detection, and two-phase commit (Section 3.2). Partial failure is handled by bringing all nodes down, Informix Data Skip (run on live nodes with no well-defined semantics), or redundancy from full failover to chained declustering, under which the n-1 survivors each take n/(n-1) of the load.

8. Q: What extra machinery does shared disk need that shared memory does not, and what failure can it still not mask? A: Because each machine has its own buffer pool and locks with no shared memory to coordinate them, shared disk needs a distributed lock manager and a cache-coherency protocol, which are complex and can bottleneck under contention (Section 3.3). It cannot mask corruption that occurs before a page reaches storage, since all nodes then see the corrupt page and RAID replicates it.

9. Q: List the rewriter's responsibilities and give an example of each. A: View expansion (replace a view with its tables and predicates recursively), constant arithmetic evaluation (`10+2+R.y` becomes `12+R.y`), logical predicate rewriting (`NOT salary > 1000000` becomes `salary <= 1000000`, and unsatisfiable conjunctions become FALSE), semantic optimization using constraints (redundant join elimination via a non-null foreign key), and subquery flattening/normalization (Section 4.2). The rewriter uses only the query and catalog, never table data.

10. Q: What did the System R/Selinger optimizer do, and how have modern optimizers extended it? A: It optimized each SELECT-FROM-WHERE block with bottom-up dynamic programming, restricted to left-deep plans, postponed Cartesian products, and used simple cardinality-based selectivity (Section 4.3). Modern systems add bushy plans, histogram and sampling-based selectivity that handles column dependencies, top-down Cascades search as an alternative to dynamic programming, two-phase parallel optimization, and plan caching with recompilation. Both DP and top-down search remain exponential in the number of tables.

11. Q: Why does an early selectivity error matter more than a late one? A: Ioannidis and Christodoulakis showed that selectivity errors early in optimization propagate multiplicatively up the plan tree, so a mistake near the leaves corrupts every estimate above it and can produce a badly wrong plan (Section 4.3, reference [45]).

12. Q: Explain the iterator model and the significance of coupling dataflow with control flow. A: Every operator subclasses an iterator with init(), get_next(), and close(), and any iterator can feed any other, so operator logic is independent of neighbors (Section 4.4). Because get_next() is an ordinary call returning a tuple via the call stack, a tuple reaches the parent exactly when control returns, so one thread runs the entire graph with no inter-operator queues or rate matching.

13. Q: How is parallelism added to the iterator model without rewriting operators? A: Parallelism and network communication are encapsulated in special exchange iterators (Graefe, reference [23]) that push data internally between partitions while still exposing the pull-style get_next() API to ordinary iterators, so the rest of the operators are unchanged (Section 4.4.1).

14. Q: What is the Halloween problem and how is it avoided? A: When a statement reads and updates the same table, pipelining an index scan into an update (e.g., a 10% raise for everyone under $20K) can rediscover a tuple that moved rightward in the B+-tree after being updated and update it repeatedly, violating SQL's rule that a statement cannot see its own updates (Section 4.4.3). Fixes: have the optimizer avoid indexes on the updated column, or materialize all target Record-IDs first and then fetch and update them; pipelined updates need multiversion storage. It is unrelated to the phantom problem.

15. Q: What is a SARG and why push it into the access method layer? A: A SARG is a search argument of the form column-operator-constant passed to an access method's init() (NULL means scan all); it is a System R term (Section 4.5). Pushing it down lets B+-trees work efficiently and, even for heap scans, lets the access method test predicates a page at a time, avoiding the per-tuple cost of returning a pinned tuple or copy, having the caller reject it, then unpinning or deleting and re-calling. This amortizes work across a collection for CPU savings while keeping a clean storage/relational boundary.

16. Q: Why do some vendors re-optimize cached plans aggressively while others do not? A: It reflects customer base. IBM served high-budget shops with skilled DBAs who value predictable performance, so it re-optimizes only when a plan can no longer run (e.g., a dropped index). Microsoft entered at the low end where customers want self-tuning, so SQL Server re-optimizes on significant cardinality changes (Section 4.3.1). The trade-off is predictability versus efficiency in a changing environment.

## 7. Glossary terms introduced

- DBMS client: the software component implementing the application API (JDBC, ODBC, OLE/DB) that marshals calls to the server (Section 2).
- DBMS worker: the thread of execution doing work for one client; 1:1 with a client (Section 2).
- OS process: an execution unit with a private address space, resource handles, and security context, scheduled by the kernel (Section 2).
- OS thread (k-thread): a kernel-scheduled execution unit sharing its process's address space, with no private OS context (Section 2).
- Lightweight thread package: an application-level, user-space-scheduled thread system; faster switches but one blocking call blocks all threads (Section 2).
- DBMS thread: a DBMS's own lightweight thread package (Section 2).
- Process per worker / thread per worker / process pool: the three baseline mappings of workers onto OS processes or threads (Section 2.1).
- Buffer pool: the shared in-memory staging area for all persistent database pages (Section 2.1.4).
- Log tail: the in-memory FIFO queue of log records periodically flushed to the log disk(s) (Section 2.1.4).
- Group commit: batching multiple commit log records into a single I/O to reduce log flush cost (Section 2.1.4).
- Lock table: the shared structure the lock manager uses to implement locking semantics (Section 2.1.4).
- Admission control: a policy that refuses new work unless resources are available, giving graceful degradation (Section 2.4).
- Intra-query parallelism: speeding up a single query by temporarily assigning multiple workers to it (Sections 2.3, 4.3).
- Shared memory / shared nothing / shared disk: the three parallel hardware models (Section 3).
- Horizontal partitioning: slicing tables by tuple across machines (hash, range, round-robin, or hybrid) in shared nothing (Section 3.2).
- Chained declustering: a fine-grain redundancy scheme spreading tuple copies so survivors share load with linear degradation (Section 3.2).
- NUMA: a shared-memory programming model over independent memories with non-uniform (local-fast, remote-slow) access times (Section 3.4).
- Pod: a subset of CPUs plus its local memory inside a large NUMA shared-memory machine (Section 3.4).
- Thread migration: reassigning a DBMS thread to a different OS process, e.g., for load balancing (Sections 2.3, 3.5).
- Four-part name: the canonical table name server.database.schema.table (Section 4.1).
- Query rewrite (rewriter): the module that simplifies and normalizes a query using only the query and catalog, not table data (Section 4.2).
- View expansion: replacing view references with the view's underlying tables and predicates, recursively (Section 4.2).
- Redundant join elimination: dropping a foreign-key-referenced table from a query when its columns are unused (Section 4.2).
- Subquery flattening: rewriting nested or correlated subqueries into a single block to expose optimizations (Section 4.2).
- System R / Selinger optimizer: the bottom-up dynamic-programming, left-deep, Cartesian-product-postponing optimizer that is the basis of cost-based optimization (Section 4.3, reference [79]).
- Selectivity estimation: predicting how many tuples a predicate passes, via cardinalities, histograms, or sampling (Section 4.3).
- Left-deep plan: a join tree whose right input is always a base table (Section 4.3).
- Cascades / top-down search: a goal-directed optimizer search strategy alternative to dynamic programming (Section 4.3, reference [25]).
- Query plan cache: the store of compiled plans reused for repeated or similar dynamic statements (Section 4.3.1).
- Iterator model: the execution model where each operator implements init/get_next/close and any operator can feed any other (Section 4.4).
- Exchange iterator: an iterator that hides parallelism and network communication behind the pull-style get_next() API (Section 4.4.1, reference [23]).
- BP-tuple / M-tuple: an in-flight tuple residing in a buffer pool page (pin-counted) versus one copied onto the memory heap (Section 4.4.2).
- Pin count: the count of active references to tuples on a buffer pool page, incremented while referenced to prevent replacement (Section 4.4.2).
- Halloween problem: repeated updates caused by a plan rediscovering its own already-modified tuples via an index scan (Section 4.4.3).
- Access method: a routine managing a disk-based data structure (heap or index) and exposing an iterator API (Section 4.5).
- SARG (search argument): a column-operator-constant predicate pushed into the access method's init() (Section 4.5).
- RID (row ID): a physical disk address used to point at base-table rows from indexes (Section 4.5).
- GiST (Generalized Search Tree): PostgreSQL's extensible index used to implement R-trees and RD-trees (Section 4.5, reference [39]).
